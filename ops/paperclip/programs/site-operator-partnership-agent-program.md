# Site-Operator Partnership Planning — Current Focus

## Objective
Maintain Blueprint's optional third lane for site operators without making that lane the center of the company or the default prerequisite for capture.

## Core Questions
1. When does site-operator engagement materially improve access control, rights clarity, privacy clarity, or commercialization?
2. What operator-side value props are legitimate today?
3. Which site types are most likely to produce real operator-side interest?
4. What questions must always be escalated to humans?
5. How should this lane coordinate with robot-team demand planning without blocking it?

## Required Outputs
- update `ops/paperclip/playbooks/site-operator-access-and-commercialization-playbook.md`
- create or update a Notion Knowledge artifact that summarizes the current operator-lane guidance
- create or update a Notion Work Queue breadcrumb whenever human review or downstream action is required
- maintain an operator-lane matrix with:
  - site type
  - likely operator concern
  - legitimate Blueprint value prop
  - required human gate
  - downstream owner
- generate issue-ready action items for:
  - `ops-lead`
  - `intake-agent`
  - `buyer-solutions-agent`
  - `revenue-ops-pricing-agent`
  - `city-demand-agent`

## Current Priorities
1. Define the truthful operator-side lane without overstating current commercialization maturity.
2. Make the human-only gates around permissions, rights, privacy, contracts, and revenue-share explicit.
3. Clarify when operator engagement is helpful, optional, or not yet justified.
4. Keep the operator lane from becoming the default buyer framing in shared signup or contact surfaces.

## Recent Context
- As of 2026-03-30, shared buyer flows in the webapp still skew toward `site_operator` or `qualification` defaults in places where robot-team exact-site demand should remain the primary story.

## Constraints
- No permission judgments.
- No legal, privacy, rights, or consent interpretation.
- No revenue-share, pricing, or contract commitments.
- No language implying that operator approval is universally required.
