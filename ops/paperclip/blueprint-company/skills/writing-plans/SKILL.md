# Writing Plans

Use this skill for non-trivial engineering work that needs an explicit plan before implementation.

## Process

1. Read the issue, requirements, and current repo context first.
2. Map the files, systems, and validation points that are likely to change.
3. Break the work into small, ordered tasks with one clear outcome each.
4. Include verification for every meaningful task.
5. Save the plan under `docs/superpowers/plans/` when the work warrants a durable artifact.

## Guardrails

- Do not start coding before the task breakdown is coherent.
- Avoid vague tasks like "implement feature"; make the steps concrete.
- If the change is trivial, keep the plan short instead of fabricating ceremony.
