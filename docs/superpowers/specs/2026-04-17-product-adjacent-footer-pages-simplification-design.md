# Product-Adjacent Footer Pages Simplification Design

Date: 2026-04-17

Owner: `webapp-codex`

Status: Approved for implementation by user-directed autonomy

## Goal

Simplify the remaining dense product-adjacent footer pages so they match the new public-site standard:

- `/sample-deliverables`
- `/proof`
- `/for-robot-integrators`
- `/for-site-operators`

These pages should become shorter, more visual, more editorial, and easier to scan without losing product truth.

## Shared Rules

- keep exact-site framing explicit
- keep public proof vs illustrative preview vs request-scoped access explicit
- keep rights, privacy, provenance, and hosted/package distinctions truthful
- remove repeated explanatory blocks and stacked card grids

## Page Directions

### Sample Deliverables

Current issue:
- too many sections repeat the same package/hosted/export story
- sample manifest, rights sheet, export bundle, package contents, hosted outputs, technical reference, stable contract, listing variation, and sample eval path all stack into a handbook

New direction:
- hero
- one proof-led sample surface section
- one combined sample contract section
- one package vs hosted deliverables section
- one closing CTA/download strip

Keep:
- sample manifest
- rights sheet
- export bundle framing
- downloads

Remove or merge:
- duplicate package/hosted lists
- duplicate input/output contract layer
- duplicate stable contract / listing variation restatements

### Proof

Current issue:
- already closer to the target, but still splits the proof story into too many medium-value sections

New direction:
- hero
- city-specific proof block when present
- proof hub / route strip
- one compact proof + package/hosted/trust section
- closing CTA

Keep:
- city-specific proof messaging
- public proof module
- proof routes

Compress:
- separate listing/runtime image section and package/hosted/trust section into a tighter main body

### For Robot Integrators

Current issue:
- persona page still reads like an old feature marketing page
- experiment-driven order and extra use-case density add unnecessary complexity

New direction:
- hero
- one exact-site value section
- one compact use-case section
- one what you get / what it does not do section
- closing CTA

Remove:
- experiment ordering logic
- redundant “what this is” vs “what you get” layering

### For Site Operators

Current issue:
- too many separate blocks for benefits, revenue highlight, how it works, control, facility types, CTAs

New direction:
- hero
- one benefits / value strip
- one “how it works + what you control” section
- one compact eligibility / facility-types section
- closing CTA

Keep:
- revenue-share framing
- access/privacy/control framing
- operator-facing next step

## Acceptance Criteria

- each page is materially shorter
- each page reaches the core proof or persona value proposition faster
- no page reads like a handbook
- all four pages remain doctrinally correct and commercially truthful
